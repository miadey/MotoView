# Bzzz on MotoView

**Bzzz** is a social super-app — Discord-style servers/channels, an X-style
feed, a full Discourse-style forum, and WhatsApp-style encrypted messaging +
status — rebuilt natively on **[MotoView](../../README.md)** (Motoko + `.mview`,
server-driven UI) and running entirely as a single Internet Computer canister.
It is a port of the C#/Wasp `bzzz` app onto MotoView, dogfooding the framework.

> Status: builds clean (`moc --check`, 0 errors) and is **verified end-to-end on
> a local replica** — all 15 routes render and the interactive flows (claim
> handle, create topic, reply, like, post to feed, create server, send DM)
> mutate shared on-canister state and persist. No fakes.

## Features

- **Internet Identity only, one-click onboarding:** every route except
  `/welcome` requires a signed-in, handle-bound II user (layout-level gate).
  The first authenticated load auto-claims a handle derived from the caller's
  II principal, so a brand-new user is **one click** ("Sign in with Internet
  Identity" + the II ceremony) away from the app; the dashboard then offers a
  one-field rename (handles are bind-or-update, changeable any time on `/me`).
- **Dashboard (`/`):** the latest from every sub-app — feed posts, forum
  topics by pure recency, per-server latest channel message, the caller's
  conversations (metadata only: peer, time, unread — message bodies are never
  shown on the dashboard), live statuses — plus member/server/topic/post
  counts and a chat-presence count.
- **Servers (Discord):** servers/guilds, channels, roles (Owner/Admin/Mod),
  moderation (pin/lock/mute), membership, messages with reactions, edits,
  replies and threads, typing/presence. — `Servers`, `Chat`
- **Forum (Discourse):** categories with the exact forum.dfinity.org colors,
  topics with tags/pinned/closed/solved, posts & replies, likes, accepted
  answers, Latest/New/Top/Categories tabs. — `Forum`
- **Feed (X):** posts, follows, reposts, likes, replies, home timeline. — `Feed`
- **Messages (WhatsApp):** 1:1 + group conversations with read receipts and
  typing; messages travel in a ciphertext-envelope field and a public key
  directory (IK/SPK/OTPK per device) exists for client-side X25519/AEAD —
  but the browser client does **not** encrypt yet, so bodies are stored as
  plaintext today (honest limitation). — `Messenger`, `Keys`
- **Status:** 24-hour ephemeral colored statuses. — `Status`
- **Identity & Admin:** handle/avatar/bio directory; admin allowlist. — `Identity`, `Admin`
- **PWA + native shells:** installable web PWA (manifest + service worker served
  by the canister) plus Tauri (desktop) and Capacitor (mobile) shells — see
  [`clients/`](clients/README.md).

## Architecture

Server-driven UI, no application JavaScript. Pages are `.mview` (HTML + Motoko
`@code`); the Rust `motoview` compiler emits one Motoko actor; the Motoko
runtime serves HTTP from the canister. State lives in **stateful services**
(`src/Services/*.mo`, each a `public class` the compiler instantiates once at
actor scope and shares across every page). See [`BUILD_SPEC.md`](BUILD_SPEC.md)
for the full authoring contract and data model.

```
src/
  Layouts/AppLayout.mview          top-nav shell + base CSS + PWA wiring
  Pages/*.mview                    15 routable pages (see BUILD_SPEC PART 2)
  Services/*.mo                    9 stateful services (Identity/Chat/Servers/
                                   Forum/Feed/Messenger/Status/Keys/Admin)
  main.mo                          GENERATED by `motoview build`
clients/                          Tauri (desktop) + Capacitor (mobile) + PWA
```

## Build & run (local)

From the repo root:

```bash
# 1. build the compiler (once)
cargo build --release --manifest-path compiler/Cargo.toml

# 2. compile the .mview app -> src/main.mo
compiler/target/release/motoview build apps/bzzz --name bzzz

# 3. type-check (optional)
~/.cache/dfinity/versions/0.28.0/moc --check \
  --package base ~/.cache/dfinity/versions/0.28.0/base \
  --package motoview runtime/src apps/bzzz/src/main.mo

# 4. deploy to a local replica
cd apps/bzzz && dfx start --clean --background && dfx deploy
# open http://<canister-id>.localhost:4955/
```

`motoview dev apps/bzzz` does steps 2 + 4 together.

## Notes & honest limitations

- **Persistence:** shared-service state lives for the canister's running
  lifetime. Upgrade-stable persistence is the framework's existing roadmap item.
- **Realtime:** chat/feed update via MotoView's adaptive polling (no WebSockets).
- **E2EE:** the canister implements the key directory + ciphertext-envelope
  storage honestly; the actual X25519 + AEAD encryption is the browser client's
  job (the canister never sees plaintext). The in-app composer currently stores
  the text as the envelope as a placeholder for that client-side crypto step.
- **Auth:** route handlers are scoped to `ctx.caller`; full Internet Identity
  login over HTTP is a MotoView roadmap item.

Why MotoView

One language, end to end

No Node, no npm, no React, no agent-js glue, no duplicated validation. Just Motoko and dfx.

Write Motoko

Template markup and Motoko code live together in one .mview file — a Blazor / Blade / LiveView feel.

Secure forms by default

The secure attribute mints a signed HMAC-SHA256 token binding path, handler, caller, nonce and expiry. Replays and tampering are rejected server-side.

SEO-friendly SSR

First load returns server-rendered HTML inside #mv-root — real markup crawlers and users see instantly, not a blank shell.

No frontend JavaScript

You ship zero application JavaScript. The only browser code is a Rust-compiled WASM client plus a tiny hand-written JS glue.

Server-driven

Events POST to the canister, run typed Motoko handlers, and return a new versioned batch. Adaptive polling synchronizes the DOM.

Rust → WASM client

The client "brain" — polling state machine, protocol, batch interpretation — is Rust compiled to WebAssembly, served from your canister.

Code beside the UI

Markup and logic in one file

The clicks are updates, the render is a query. State lives in @code right next to the buttons that change it — and persists across calls in your canister.

  • Bake handler args server-side with @click="increment(1)"
  • Interpolate state inline with @count
  • State persists in the canister across calls
  • Compiles straight to a Motoko actor
src/Pages/Counter.mview 
@page "/"
@layout MainLayout
@title "Counter"

<section class="mv-container">
  <h1>Counter</h1>

  <p>Current value: <strong>@count</strong></p>

  <button @click="increment(1)">+1</button>
  <button @click="increment(5)">+5</button>
  <button @click="decrement">-1</button>
  <button @click="reset">Reset</button>
</section>

@code {
  var count : Nat = 0;

  func increment(by : Nat) : async () {
    count += by;
  };

  func decrement() : async () {
    if (count > 0) { count -= 1 };
  };

  func reset() : async () {
    count := 0;
  };
}
